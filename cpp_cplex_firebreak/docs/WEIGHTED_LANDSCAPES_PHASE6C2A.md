# Weighted Landscapes Phase 6C2A: Integer Combinatorial Lifting

Phase 6C2A audits and enables the existing integer-incumbent FPP
combinatorial lifting modes for arbitrary positive landscape weights. The scope
is deliberately limited to cuts generated by the Phase 6C1 integer callback
separator. It does not add fractional cuts, root combinatorial cuts, initial
cuts, scenario sampling, eta-desc ordering, restricted-candidate combinatorial
Benders, DPV, Static-DPV, manifests, or batch method changes.

## Audited Modes

The repository exposes exactly three FPP combinatorial lift modes:

```text
none
posterior
heuristic
```

There are no `exact` or `sequential` CLI/parser modes in the current codebase.
Both `posterior` and `heuristic` are implemented in
`FppCombinatorialBendersSeparator::separateScenario` as path-level candidate
deduplication after the valid Phase 6C1 weighted path-activation cut is built.
They do not solve auxiliary lifting subproblems, do not run extra wildfire
propagation, and do not use candidate scores. The names are legacy option
labels; in the current implementation they are not heuristic coefficient
estimators.

Mode classification:

```text
none       validated baseline weighted path-activation cut
posterior  valid conservative exact path-dedup lifting
heuristic  valid conservative exact path-dedup lifting under a legacy name
```

## Baseline Cut

For scenario `s`, incumbent `ybar`, active burned destinations
`A_s(ybar)`, selected activation path `P_{s,k}` for destination `k`, and
positive destination weight `w_k`, the Phase 6C1 cut is:

```text
eta_s >= gamma_s + sum_i alpha_i y_i

gamma_s = sum_{k in A_s(ybar)} w_k
alpha_i = - sum_{k: i appears on P_{s,k}} w_k
```

Loss-valued terms use destination-cell weights. Candidate weights are never used
as a proxy for protected value.

## Weighted Path-Dedup Lifting

The supported Phase 6C2A transformation changes only the path incidence count
inside each destination term:

```text
alpha_i^lift = - sum_{k: i in unique(P_{s,k})} w_k
```

The constant remains:

```text
gamma_s = sum_{k in A_s(ybar)} w_k
```

For simple parent paths this is identical to the baseline cut. If a candidate
appears more than once in a stored path, deduplication makes the coefficient
less negative while preserving validity:

```text
w_k * (1 - sum_{i in unique(P_{s,k})} y_i) <= w_k * x_k(y)
```

If no candidate on the path is selected, the path remains open and `k` burns.
If at least one candidate on the path is selected, the term is non-positive and
therefore lower-bounds the nonnegative burn indicator. Summing over destinations
gives global validity for all binary feasible `y` under the firebreak budget.

The transformation is scenario-wise, so it is compatible with expected value,
CVaR, and mean-CVaR masters without risk-specific formulas.

## Incumbent Tightness and Dominance

At the generating integer incumbent, every active path has no selected non-root
candidate on it. The lifted RHS therefore remains:

```text
LiftedRHS_s(ybar) = gamma_s = Q_s(ybar)
```

The lifted cut dominates or equals the baseline over the feasible binary domain
because every lifted coefficient is greater than or equal to the corresponding
baseline coefficient and the constant is unchanged. If the implementation ever
detects a lifted coefficient weaker than the baseline dominance check allows, it
throws before adding the cut.

## Candidate Universe and Deduplication

The separator uses the compact eligible candidate universe from the optimization
instance. Original Cell2Fire IDs remain available through existing result
metadata. The cut signature already includes scenario, cut type, constant,
complete sparse coefficient vector, weight-map hash, and candidate universe, so
baseline and lifted cuts with different coefficients are not merged
accidentally.

There is no lifting cache in Phase 6C2A. If a future cache is added, it must key
on scenario ID, weight-map hash, candidate universe, budget, generating
incumbent, baseline cut identity, lifting mode, lifting order, and
implementation version.

## Supported Non-Homogeneous Configuration

For non-homogeneous maps, Phase 6C2A allows only:

```text
enabled=true
lift_mode=none|heuristic|posterior
scenario_order=eta-asc
cut_sampling_ratio=1
separate_fractional=false
initial_cuts=false
```

The following remain rejected for non-homogeneous combinatorial runs:

- fractional combinatorial cuts;
- root combinatorial cuts;
- initial combinatorial cuts;
- scenario sampling below the full training set;
- eta-desc ordering;
- LLBI/projected LLBI combinations with combinatorial Benders;
- global dominance and conditional zero-benefit fixing combinations with
  combinatorial Benders;
- restricted-candidate combinatorial Benders;
- DPV and Static-DPV combinatorial variants.
